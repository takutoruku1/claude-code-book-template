# UIワイヤーフレーム レビュー（UI設計視点）

> レビュー日: 2026-05-10  
> レビュー対象: `docs/UIワイヤーフレーム.html`  
> レビュー観点: UI設計書（`docs/UI設計書.md`）との整合性

---

## 総合評価

**B — 骨格は設計書に沿っているが、カラートークンの乖離と設計書未定義コンポーネントが複数あり、エンジニアへの正確な仕様伝達には修正が必要。**

---

## 重大な問題点（修正必須）

1. **カラートークンの定義値が設計書（css/main.css）と全面的に相違している**

   UI設計書が規定する正規のトークン値と、ワイヤーフレームで定義された値が全く異なる。

   | トークン | UI設計書（main.css） | ワイヤーフレーム（UIワイヤーフレーム.html） |
   |---------|---------------------|--------------------------------------------|
   | `--bg` | `#0d0d14` | `#07070c` |
   | `--surface` | `#16161f` | `#0e0e16` |
   | `--surface2` | `#1e1e2e` | `#13131e` |
   | `--border` | `#2a2a40` | `#1a1a2a` |
   | `--text` | `#e8e8f0` | `#dae0f0` |
   | `--text-mid` | 設計書に定義なし（`--text-dim: #888899`のみ） | `#6878a0`（独自定義） |
   | `--text-dim` | `#888899` | `#3d4a6a` |

   ワイヤーフレームは全体的にさらに暗い色調で設計されており、コントラスト比計算や視覚確認の基準がずれる。アクセシビリティ検証（AA合格を設計書8章で確認済みの値）も意味をなさなくなる。

2. **設計書に定義されていないカラー変数 `--text-mid: #6878a0` の独自導入**

   設計書のテキストカラーは `--text: #e8e8f0` と `--text-dim: #888899` の2段階。ワイヤーフレームでは中間色として `--text-mid` を追加定義しているが、設計書には存在しない。エンジニアがこの値をmain.cssに追加すべきかどうか判断できない。設計書への記載または削除が必要。

3. **プレイヤーバブルの背景が設計書仕様と異なる**

   設計書5-1では `.msg-bubble.self` の背景を `var(--chat-self)` = `#2a1f4a` と規定しているが、ワイヤーフレームでは `rgba(124,106,247,.10)` を直接指定している。設計書に定義された機能カラー変数 `--chat-self` を使用すべきである。

4. **クライアントバブルの背景が設計書仕様と異なる**

   設計書5-1では `.msg-bubble.client` の背景を `var(--chat-other)` = `#1e2a1e` と規定しているが、ワイヤーフレームでは `rgba(6,199,85,.08)` を直接指定している。機能カラー変数 `--chat-other` の使用に修正が必要。

5. **タイトルロゴのフォントサイズが設計書の規定と異なる**

   設計書2-2のフォントサイズ体系では「タイトルロゴ（新デザイン）」を `clamp(52px, 9vw, 86px)` と規定しているが、ワイヤーフレームでは `clamp(36px, 8vw, 64px)` を使用している。最小値・最大値ともに設計書規定を大幅に下回る。

6. **設計書に定義のないカラー変数（`--saku`, `--seiji`, `--karen`, `--midori`, `--blue`, `--green`, `--purple`, `--red`, `--yellow`）が独自定義されている**

   ワイヤーフレームの `:root` に設計書には存在しない多数のトークンが定義されている。これらを正規のカラートークンとしてmain.cssに追加するのか、ワイヤーフレーム専用の仮置きなのかが不明瞭である。特にキャラクター色（`--saku`, `--seiji`, `--karen`）については設計書6章で言及はあるがCSS変数としての正式定義が設計書にない。明示的な注記が必要。

## 改善推奨事項

1. **選択肢ボタンのhoverトランジションが設計書と異なる**

   設計書5-2では `.choice-btn:hover { transform: translateX(3px); }` という右方向スライドを規定しているが、ワイヤーフレームはその仕様を持たない。ワイヤーフレームはデザイン確認ドキュメントとして機能することを想定しているため、インタラクション動作も設計書に合わせるべきである。

2. **スコアバーのトランジション定義が設計書と異なる**

   設計書5-6では `transition: width .5s cubic-bezier(.34,1.56,.64,1);` を規定しているが、ワイヤーフレームでは `transition: width .8s ease;` となっている。イージング関数が異なるため、エンジニアが実装時に混乱する。

3. **素材カードの角丸（border-radius）が設計書の規定と異なる**

   設計書2-3では素材カードの角丸を `14px` と規定しているが、ワイヤーフレームでは `8px` が使われている。設計書のコンポーネント定義（5-3の `.card-face` で `border-radius: 14px`）にも反する。

4. **選択肢ボタンの角丸（border-radius）が設計書の規定と異なる**

   設計書2-3では選択肢ボタンの角丸を `8px` と規定し、5-2でも同様に `border-radius: 8px` と定義しているが、ワイヤーフレームでは `6px` を使用している。

5. **ハッシュタグ（投稿プレビュー）の色がUI設計書のトークンと異なる**

   設計書5-4では `preview-hash` に `color: var(--buzz-blue)` = `#1d9bf0` を規定しているが、ワイヤーフレームでは `color: var(--blue)` = `#38bdf8` を独自定義して使用している。色温度が異なるため視覚的な印象が変わる。

6. **反響フィード（.feed-card / .posted-card）の角丸が設計書と異なる**

   設計書5-5では `feed-card { border-radius: 14px; }` を規定しているが、ワイヤーフレームの `.posted-card` は `border-radius: 10px` を使用している。

7. **エンディング画面の背景グラデーションが設計書の規定と異なる**

   設計書4-7では `#screen-ending` の背景を `radial-gradient(ellipse at center, #1a0a2e 0%, #0d0d14 70%)` と規定しているが、ワイヤーフレームでは `linear-gradient(180deg, #0a0a14 0%, #1a0a2e 100%)` となっており、グラデーション種別（radial vs linear）も方向も異なる。

8. **エンディング種別の数が設計書と不一致**

   設計書4-7ではエンディングを8種と記述しているが（`継続エンド` を含む）、ワイヤーフレームでは7種しか示されていない（`継続エンド` が欠落している）。設計書の画面一覧の「8種」との整合を確認・修正すること。

## 軽微な指摘

1. **アノテーション内のバブルスタイル記述が実際の実装値と一致していない**

   チャット画面のアノテーションブロック（行1326-1331）では「プレイヤー: 背景 rgba(accent2,.10)」と記載しているが、`accent2` という変数名は設計書定義のトークン `--accent2: #7c6af7` を指すものの、ワイヤーフレームのCSS実装では `rgba(124,106,247,.10)` と数値で直接指定されており表記が不統一。

2. **タイパーアニメーションの速度が設計書の規定と異なる**

   設計書4-1では「表示速度は44〜82ms/文字」と規定しているが、ワイヤーフレームのJSでは `55 + Math.random() * 35` ms（55〜90ms/文字）を使用している。下限・上限ともに設計書規定と異なる。

3. **フラッシュバックバブルのスタイルが設計書の仕様と異なる**

   設計書5-1では `protag-bubble.flashback { background: rgba(12,8,24,.96); }` と規定しているが、ワイヤーフレームでは `background: rgba(0,0,0,.4)` と半透明度が大きく異なる。

4. **フォントファミリーの定義が設計書と異なる**

   設計書2-2では `body { font-family: 'Noto Sans JP', sans-serif; }` を規定し、Google Fontsのインポートを含む。ワイヤーフレームでは `font-family: 'Hiragino Kaku Gothic ProN', 'Noto Sans JP', 'Yu Gothic', sans-serif;` と記述しており、Noto Sans JPを優先していない。ワイヤーフレームとしての確認用途であれば許容範囲だが、実装時の混乱を避けるため注記が望ましい。

5. **タイトル画面のウィンドウ幅が設計書と異なる**

   設計書4-1ではタイトルウィンドウ幅を `min(660px, 92vw)` と規定しているが、ワイヤーフレームでは `min(480px, 90vw)` を使用しており、最大幅が180px狭い。

6. **`switchEnding`関数内の未使用コード**

   ワイヤーフレームのJS（行2046）に `document.querySelectorAll('.ending-mock').length; // no-op reference` という意味のないコードが残っており、品質面で気になる。

## 良かった点

1. **7画面すべてを網羅したタブ構成**

   設計書で定義された全ゲーム画面（タイトル・チャット・クイズ・素材確認・投稿ビルダー・反響フェーズ・エンディング）を一つのHTMLファイルでタブ切り替えにより確認できる構成は、ステークホルダーへのレビュー提示に優れている。

2. **アノテーションシステムの充実**

   `annot`・`annot-block`・`wf-note`・`spec-list` の4段階のアノテーション表記により、「ラベル付け」「詳細仕様」「実装ノート」「箇条書き仕様」を視覚的に区別できている。設計書との橋渡し資料として仕様伝達力が高い。

3. **インタラクティブなデモ操作の実装**

   選択肢ボタンの選択・クイズの正誤判定・エンディング種別切り替えなど、主要なインタラクションをデモ操作で確認できる。静的なワイヤーフレームと比較して動作確認の価値が高い。

4. **エンディング7種の分岐条件をワイヤーフレーム内に明記**

   エンディング画面パネルのアノテーションブロックにスコア条件（selfScore/buzzScore/egoScore）を数値で明記しており、設計書4-7の内容と対応した正確な仕様記載になっている。

5. **キャラクター別カラーの視覚的区別**

   素材カードのmysteryパープル枠、チャットバブルのクライアント色（みどり系）・プレイヤー色（パープル系）など、設計書6章で定義されたキャラクター別カラー識別の考え方を正しくビジュアルに反映している。

6. **レスポンシブ用メディアクエリの記述**

   チャット・ビルダー・二列レイアウト等に `max-width: 600px / 640px / 560px` のブレークポイントを設けており、設計書7章の「モバイル対応は最小限だが考慮する」方針に沿った対応がなされている。
