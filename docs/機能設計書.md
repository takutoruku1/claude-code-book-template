## 1. プロジェクト概要

### ゲームコンセプト・テーマ

「バズったー」はSNSコンサルタントを主人公にしたナラティブゲーム（ビジュアルノベル風）。プレイヤーは元ライターの主人公・瀬川 湊（せがわ みなと）として、4人のクライアントのSNS代行を行う。

**主人公の背景**: 3年前、バズ目的の告発記事を書き、対象者（篠宮カレン）がSNSを削除して消息不明になった。以来ライター業を断ち、「その人らしい言葉を守る仕事」として SNS 運用代行を行っている。

**ゲームの核心テーマ**: 「バズ（数字）」と「らしさ（自己表現）」のトレードオフ、言葉の暴力と翻訳者としての贖罪。

### ターゲットユーザー

- **年齢層**: 20〜35歳の社会人
- **想定プレイ時間**: 1ルートあたり30〜45分

### ゲームジャンル・形式

- ジャンル: ナラティブゲーム（ビジュアルノベル風）
- 技術形式: Vanilla HTML/CSS/JS（単一HTMLファイルベース + JSモジュール分割）
- UI形式: スマートフォン OS 風デスクトップ UI（ウィンドウ管理・タスクバー）

```
  protagonist.js           # 主人公ウィジェット（内部モノローグ表示）
  save-manager.js          # セーブ/ロード（3スロット）・shutdownGame()・loadFromSlot()

  y-window.js              # 「Y（X/Twitter風）」ウィンドウ制御
```

### モジュール間の依存関係

```
app-core.js
  └─ CHARACTERS, GS, initGS(), applyBonus(), buildFlowMap()
       ├─ chat-engine.js が参照（runChatId, canShow）
       ├─ game-logic.js が参照（goToPost, resolveEnding）
       └─ save-manager.js が参照（GS の読み書き）

scenario.js
  └─ CHAT_FLOWS, MATERIALS, REACTIONS, CHAT_QUIZ
       ├─ chat-engine.js が CHAT_FLOWS を実行
       └─ game-logic.js が MATERIALS/REACTIONS を参照

postOptions.js
  └─ POST_TEXTS, styleOptions, HASH_OPTIONS_BY_ROUTE, getHashOptions()
       └─ game-logic.js が goToPost() / buildReactionFeed() で呼び出し

endings.js
  └─ ENDINGS（8エンド定義）
       └─ game-logic.js の showEnding() が参照
```

## 3. ゲームステート管理

### GS（GameState）オブジェクトの全変数定義

`initGS(route)` によって初期化される（`js/app-core.js` 内）。

```javascript
let GS = {
  // ルート
  route: String,          // 'midori' | 'saku' | 'seiji' | 'karen'

  // スコア（最終計算値）
  buzz: 0,                // Number: バズ度（0〜100）。Math.min/max でクランプ
  self: 0,                // Number: 自己表現度（0〜100）。Math.min/max でクランプ

  // チャットフェーズで蓄積するボーナス基礎値
  chatBuzzBonus: 0,       // Number: 選択肢でのバズボーナス蓄積値
  chatSelfBonus: 0,       // Number: 選択肢のらしさボーナス蓄積値（クリップでも+8加算）

  // 投稿フェーズ開始時に chatBuzz/SelfBonus からコピーされる計算用ベース
  chatBuzzBase: 0,        // Number: goToPost() で chatBuzzBonus からコピー
  chatSelfBase: 0,        // Number: goToPost() で chatSelfBonus からコピー

  // 主人公の業スコア
  egoScore: 0,            // Number: +1（感情を出す選択）/ -1（仕事に徹する選択）で変動
                          //         正: 贖罪エンド / 0以下: 沈黙エンド

  // フラグ管理
  flags: {},              // Object: 任意フラグの連想配列（例: flags.kShinonoya = true）

  // ミステリー
  mysteryClues: [],       // Array<String>: カードめくりで蓄積（'karen' | 'yukiwarisou'）

  // チャット再開ポイント
  chatResumeAt: null,     // String | null: showMaterial 完了後の再開チャットID
  postResumeAt: null,     // String | null: showPost 完了後の再開チャットID

  // 投稿ビルダー選択状態
  selectedStyle: null,    // Number | null: styleOptions のインデックス（0〜3）
  selectedHash: null,     // Number | null: hashOptions のインデックス（0〜4）
  selectedTime: null,     // String | null: 'morning' | 'noon' | 'evening'（初期値: null）
  selectedImageSrc: null, // String | null: 選択した画像の src URL

  // 素材カード管理
  selectedCards: [],      // Array<Number>: 選択済み素材カードのインデックス（最大2枚）
  flippedCards: [],       // Array<Number>: めくり済み素材カードのインデックス

  // フォロワー数
  baseFollowers: 1240,    // Number: 初期フォロワー数

  // フラッシュバック進行度
  flashbackPhase: 0,      // Number: 0〜3（既読エンドには 3 が必須条件）

  // メモシステム
  collectedKeywords: [],  // Array<String>: クリップしたキーワード（重複なし）
  collectedNotes: [],     // Array<Object>: { keyword, text } の詳細メモ
};
```

### 主要フラグ一覧

| フラグ名 | 型 | 設定タイミング | 用途 |
|---|---|---|---|
| `kShinonoya` | Boolean | 朔ルート 葛藤2「アカウント名を確認」選択時 | 到達エンドの必須条件 |
| `zange` | Boolean | 花蓮ルート 謎収束「記事を書いたのは私」選択時 | 贖罪エンド方向フラグ |

### 4人のクライアント一覧

### キャラクタープロフィール詳細

#### 田中 みどり（みどりルート）
- **依頼理由**: お母さんが施設に入った。離れていても繋がっている実感が欲しい。植物と料理の写真を投稿したいが「センスがない」と思っている。
- **テーマ**: 「完璧じゃない」ことをらしさとして受け入れる
- **キーワード**: 「お母さんに届けたい」「毎朝見るのが好き」

#### 倉田 朔（朔ルート）
- **依頼理由**: 夜間工場勤務の傍ら一人でミニチュア模型を制作。「誰かに見てほしい」が動機。言語化が苦手で短文のみ。
- **ミステリー接続**: 素材の付箋に「カレンが好きそうだと思って」という名前。投稿後に `@k_shinonoya` からDMが届く。
- **キーワード**: 「なんか残したくて」「誰かに見てもらいたい」

#### 松本 誠司（誠司ルート）
- **依頼理由**: 古い商店街の喫茶店を営む。常連は高齢者ばかりで若い客層を呼びたい。インスタ映えには違和感を持つ。
- **ミステリー接続**: 素材写真の背景に「喫茶 ゆきわりそう」の看板が映り込む（朔の模型写真にも同じ看板）。
- **キーワード**: 「若い人に来てほしい」「映える店じゃない」

#### 篠宮 花蓮（花蓮ルート）
- **真の依頼理由**: 姉（篠宮カレン）が3年前SNSの記事で傷つき消えた。「言葉は怖くない」と姉に証明したい。朔と誠司から紹介されたのは偶然ではなく、花蓮が意図的に組んだリスト。
- **謎の収束**: 花蓮ルートですべての伏線（karen・yukiwarisou・k_shinonoya）が収束する。
- **キーワード**: 「姉に見ていてほしい」「過去を知っている」

```
→ 投稿・反響完了後 → karen_mystery_choice（謎収束選択肢: mysteryClues >= 2 の場合のみ表示）
→ 終端
```

k_memo をめくると 2500ms 後に `triggerFlashback(3, null)` が発火し `flashbackPhase = 3` になる。

#### エンディング到達条件（2段階判定構造）

花蓮ルートのエンディング判定は以下の**2段階構造**で動作する：

**第1段階: `resolveKarenEnding()` によるエンドキー決定（実行時に評価）**

| エンディング | 条件 |
|---|---|
| ズレエンド | buzz >= 65 かつ self < 50 |
| 既読エンド | 上記以外のすべて（`resolveKarenEnding()` のデフォルト return） |

> `resolveKarenEnding()` は buzz/self の値のみで判定し `'kidoku'` または `'zure'` を返す。self や flashbackPhase の条件はここでは一切チェックされない。

**第2段階: `endings.js` の `kidoku` エントリの `condition` フィールドについて**

`endings.js` の `condition: '花蓮ルート / selfScore >= 65 かつ flashbackPhase >= 3'` は**データの注釈コメント**にすぎず、`showEnding()` の実行時には評価されない。`showEnding()` は `endKey` を受け取って `ENDINGS[endKey]` の演出データを表示するだけであり、self や flashbackPhase の値によって kidoku への到達が阻まれることはない。

つまり「self が 65 未満でも、flashbackPhase が 3 未満でも、buzz < 65 かつ self >= 50 であれば kidoku エンドに到達できる」が現実の実装である。`condition` フィールドの数値条件はゲームデザイン上の想定値（既読エンドにふさわしい状態の目安）であり、ランタイム強制ではない。

## 6. スコアシステム

### スコア加算ルール

#### chatSelfBonus / chatBuzzBonus の加算

| イベント | 変動 |
|---|---|
| 選択肢の selfBonus を選ぶ | chatSelfBonus += 値（みどり: +5, 朔葛藤1: +10, 誠司: +5） |
| 選択肢の buzzBonus を選ぶ | chatBuzzBonus += 値（朔葛藤1: +5, 誠司: +5） |
| clippable キーワードをメモする | chatSelfBonus += 8（chat-engine.js の collectKeyword()） |

#### 投稿フェーズのスコア計算

`goToPost()` 内で `chatSelfBonus` / `chatBuzzBonus` を `chatSelfBase` / `chatBuzzBase` にコピーし、その後 `updatePreview()` で以下を合算する。

```
buzz = chatBuzzBase + styleOptions[i].buzz + hashOption[j].buzz
self = chatSelfBase + styleOptions[i].self + hashOption[j].self
buzz = Math.min(100, Math.max(0, buzz))
self = Math.min(100, Math.max(0, self))
```

#### 時間帯ボーナス（submitPost() 内でのみ適用）

| 時間帯 | self | buzz |
|---|---|---|
| morning（朝） | +10 | +5 |
| noon（昼） | +0 | +15 |
| evening（夕） | +5 | +10 |

> 時間帯ボーナスは `submitPost()` 内の `applyBonus()` 呼び出し時にのみ GS.self / GS.buzz に加算される。`updatePreview()` によるスコアバーのリアルタイム更新（時間帯ボタン選択時を含む）は時間帯ボーナスを含まない状態で計算される。UI上でスコアバーの値は選択した文体・ハッシュに基づいてリアルタイム更新されるが、時間帯ボーナス分は「投稿」ボタンを押した後に初めて反映される。

#### 反響フェーズのリアクションキー判定

`getReactionKey()` により以下のように分類：

| キー | 条件 | いいね | RT | フォロワー増減 |
|---|---|---|---|---|
| `high_self` | self >= 60 かつ buzz < 60 | 47 | 8 | +12 |
| `high_buzz` | buzz >= 65 かつ self < 50 | 318 | 102 | +234 |
| `balanced` | それ以外 | 89 | 23 | +41 |

### エンディング判定条件（resolveEnding / resolveKarenEnding）

```javascript
// みどりルート
if (self >= 65)                  → 'jiritu'（自立エンド）
if (buzz >= 65 && self < 50)    → 'zure'（ズレエンド）
else                             → 'jizoku'（継続エンド）

// 朔ルート
if (buzz >= 65 && self < 50)           → 'zure'
if (self >= 60 && flags.kShinonoya)    → 'toutatu'（到達エンド）
else                                   → 'zure'（kShinonoya がない場合もズレ）

// 誠司ルート
if (buzz >= 65 && self < 50)     → 'zure'
else                             → 'saihan'（再訪エンド）

// 花蓮ルート（resolveKarenEnding）
// ※ resolveKarenEnding() は buzz/self のみで判定。self や flashbackPhase の条件はここでは評価されない
if (buzz >= 65 && self < 50)    → 'zure'
else                            → 'kidoku'（既読エンド・デフォルト）

// 既読エンド後の分岐
lastChoice「よかった とだけ送る」→ egoScore += 1
if (egoScore > 0)               → 'zange'（贖罪エンド）
if (egoScore <= 0)              → 'chinmoku'（沈黙エンド）
```

## 7. ミステリーシステム

### mysteryClues の蓄積メカニズム

`GS.mysteryClues` は配列で、素材カードをめくった時に `handleCardClick()` が `m.mysteryFlag` を確認し、未追加の場合のみ `push` する（重複なし）。

#### mysteryClues の UI上の可視化

mysteryClues が追加される UI上のフィードバックは以下の2経路で行われる（ソースコード上で確認済み）：

1. **素材カードのタグ表示**: カードをめくった時に `card-back-tags` エリアにカードの `tags` 配列がアニメーション付きで表示される（`m.tags.forEach` で `.card-tag` 要素を生成）。mysteryFlag のあるカードの `tags` には「制作感」「謎収束」等のキーワードが含まれ、プレイヤーがミステリーの手がかりを確認する視覚的手段となる。
2. **主人公ウィジェットのモノローグバブル**: `protagonist.js` の `showProtagMsg()` により、カードめくり・チャット進行時に主人公のモノローグバブル（`#protagonistWidget` の `.protagBubble`）が一時的に表示される。karen ルートでは `_PROTAG_ON_CLIENT` の定義に「（何かを隠している）」「（依頼の裏を探る）」等が含まれ、ミステリーへの注目を促す演出として機能する。

> mysteryClues の蓄積状態（配列の中身）自体は画面上に直接表示されない。プレイヤーは素材カードをめくった際の視覚・テキストフィードバックから状況を把握する。`mysteryClues.length >= 2` の条件判定は `karen_mystery_choice` ステップの表示制御にのみ使われる。

```javascript
// game-logic.js: handleCardClick()
if (m.mysteryFlag && !GS.mysteryClues.includes(m.mysteryFlag)) {
  GS.mysteryClues.push(m.mysteryFlag);
}
```

#### 蓄積タイミング一覧

| ルート | カードID | mysteryFlag | 追加クルー |
|---|---|---|---|
| 朔 | `s_desk` | `karen` | 「カレン」という名前が付箋に |
| 朔 | `s_model` | `yukiwarisou` | 模型写真の背景に看板 |
| 誠司 | `sj_sign` | `yukiwarisou` | 喫茶店の看板（2件目） |
| 花蓮 | `k_flower` | `yukiwarisou` | 生け花写真の背景に看板 |

> `karen` クルーは朔ルートの `s_desk` のみで追加される（1件）。`yukiwarisou` は複数ルートで追加されるが配列への重複追加はない。mysteryClues.length >= 2 を満たすには朔ルートで `s_desk` と `s_model`（または他の yukiwarisou カード）の両方をめくる必要がある。

### flashbackPhase の発火条件

| フェーズ | タイミング | テキスト概要 |
|---|---|---|
| Phase 1 | ゲーム開始直後（プロローグ） | 「深夜2時。送信ボタンを押した。@k_shinonoya のアカウントが削除されました」CRT風テキスト5秒 |
| Phase 2 | 朔ルート・`saku_fb2` trigger（「残したい」発言後） | 主人公の内部モノローグ「——残す。あの記事も、残っている。」0.8秒停止後・暗赤色吹き出しで表示 |
| Phase 3 | 花蓮ルート・`k_memo` カードめくり時（2.5秒後） | 現在カードと過去記憶の2枚並列表示。チャットに「……篠宮」が出現。`flashbackPhase = 3` にセット |

> flashbackPhase の UI表現詳細（DOM構造・アニメーション仕様）は UI設計書 §5-7（フラッシュバック演出）を参照すること。

### 謎収束選択肢の出現条件

花蓮ルートの `karen_mystery_choice` ステップは以下条件を満たす場合のみ表示される：

```javascript
condition: 'mysteryClues.length >= 2'
// canShow() 内で Function コンストラクタを使って動的評価
```

条件を満たさない場合は `karen_mystery_choice` ステップ自体がスキップされ、`karen_end` へ直接進む（「花蓮の依頼の真意は、まだわからない」扱い）。

## 9. 投稿ビルダー機能

### 文体選択（styleOptions）

`postOptions.js` の `styleOptions` 配列（全ルート共通のインデックス、テキストはキャラ別）：

| インデックス | ラベル | buzz加算 | self加算 | 特記 |
|---|---|---|---|---|
| 0 | 短くシンプルに | +30 | +80 | — |
| 1 | 丁寧に感情を込めて | +50 | +70 | — |
| 2 | バズを狙って大胆に 🔥 | +85 | +15 | `isBuzz: true` |
| 3 | 失敗も含めてリアルに | +45 | +90 | — |

文体テキストはキャラクター × 選択素材で動的生成（`POST_TEXTS[route][styleIndex](selectedCardIds)`）。

### 素材カード枚数（ルート別）

| ルート | 枚数 |
|---|---|
| みどり | 4枚 |
| 朔 | 4枚 |
| 誠司 | 3枚 |
| 花蓮 | 3枚 |

> UIレイアウト（グリッド列数・カードサイズ）設計時はルート別の枚数差異を考慮すること。

### 素材選択ロジック

1. 素材カード（ルートにより3〜4枚）を1枚ずつクリックしてめくる（裏面を確認）。**めくっていないカードは選択不可**（`handleCardClick()` が1回目クリックでめくり処理を行い `return` で終了する）。
2. めくった後に再クリックで「選択」（最大2枚まで）
3. **2枚選択済みの状態で別カードをクリックしても無視される**（`if (GS.selectedCards.length >= 2) return;` による早期脱出）
4. 選択したカードに応じて `styleUnlock` / `hashUnlock` のインデックスが解放される
5. 解放されていない選択肢ボタンは `disabled` 状態（`btn.title = 'この素材を選ばないと使えません'`）
6. チャット中の画像も選択可能（`selectedImageSrc`）

#### locked 状態の仕組み（`lockExcessCards()`）

2枚選択済み状態になると `lockExcessCards()` が呼ばれ、未選択カードに `.locked` クラスが付与される。`.locked` は `opacity: 0.45; pointer-events: none` でグレーアウト表示され操作不可になる（UI設計書の `.card-flip-wrapper.locked` 参照）。選択を1枚解除すると `.locked` クラスが外れ、再び全カードが操作可能になる。

| 状態 | クラス | 操作 |
|---|---|---|
| 通常（未めくり） | なし | クリックでめくり |
| めくり済み（未選択） | `.flipped` | クリックで選択 |
| 選択済み（2枚未満） | `.flipped .selected` | クリックで選択解除 |
| ロック済み（2枚選択後・他カード） | `.flipped .locked` | 操作不可（pointer-events: none） |
| disabled（styleUnlock/hashUnlock 未解放） | `disabled` 属性 | 文体/ハッシュボタンが非活性 |

```javascript
// goToPost() 内
GS.selectedCards.forEach(ci => {
  mats[ci].styleUnlock.forEach(x => unlockedStyles.add(x));
  mats[ci].hashUnlock.forEach(x => unlockedHash.add(x));
});
```

#### styleUnlock / hashUnlock によるロック解除

各カードの `styleUnlock` / `hashUnlock` 配列は「そのカードを選択した時に解放される文体/ハッシュのインデックス」を定義する。カードを選択（`GS.selectedCards` に追加）するたびに `goToPost()` → `buildOpts()` が再計算し、解放済みインデックスのボタンのみ活性化される（詳細は各ルートの素材リスト表 §5 参照）。

### ハッシュタグ生成（キャラクター別）

`HASH_OPTIONS_BY_ROUTE` でキャラクター別に定義（全キャラ共通インデックス構造）：

| インデックス | みどり | 朔 | 誠司 | 花蓮 | buzz | self |
|---|---|---|---|---|---|---|
| 0 | #植物のある暮らし | #ジオラマ #模型制作 | #喫茶店 #純喫茶 | #生け花 #いけばな | +30 | +15 |
| 1 | #今日の一枚 | 同左 | 同左 | 同左 | +20 | +20 |
| 2 | #バズれ #フォロワー増やしたい | 同左 | 同左 | 同左 | +40 | **-30** |
| 3 | #ゆるく記録する | 同左 | 同左 | 同左 | +15 | +35 |
| 4 | タグなし | 同左 | 同左 | 同左 | -10 | +5 |

> インデックス 2 は `isBuzz: true`。バズ選択肢はカードをめくって unlockHash に 2 が含まれている場合のみ活性化（デフォルト解放は廃止済み）。

### 投稿プレビューと投稿ボタン活性化条件

`updatePreview()` は文体・ハッシュ・時間帯ボタンのいずれかがクリックされるたびに常に呼ばれ、スコアバー（buzzBar / selfBar）のリアルタイム更新を行う（時間帯ボーナスを除く）。投稿ボタンの活性化条件は `selectedStyle != null && selectedHash != null && selectedTime != null` の全て揃った時のみ。**素材カードを1枚も選択していない状態でも文体・ハッシュ・時間帯を選べば投稿ボタンは活性化される**（素材選択は投稿ボタン活性化の必須条件ではない。ただし素材未選択の場合は全 `styleUnlock` / `hashUnlock` が空になるため実際には文体・ハッシュの選択肢がすべて disabled になる）。


### セーブスロット

- キー: `buzzutter_save_1` / `buzzutter_save_2` / `buzzutter_save_3`（localStorage）
- スロット数: 3
- 上書き保存の場合は確認なしで即上書き

### セーブ/ロードのタイミング

| 操作 | 関数 | タイミング |
|---|---|---|
| セーブして終了 | `shutdownGame()` | プレイヤーがアプリを閉じる操作時 |
| セーブのみ | `sleepGame()` | スリープボタン押下時 |
| ゲーム中ロード | `showInGameLoadPicker()` | ゲーム中のロードメニュー選択時 |

| セーブ削除 | `showDeleteSlotPicker()` | タイトル画面の削除ボタン押下時 |

### ロード時の復元処理

`loadFromSlot(slot)` 内で以下を復元する：

- `GS` オブジェクト全体の上書き
- チャット履歴の DOM 再構築
- `currentArea` に応じた画面状態の復元
  - `materialArea`: `setupMaterials()` → めくり・選択状態を再適用
  - `postArea`: `_restorePostArea()` で選択状態を再適用
  - `reactionArea`: `buildReactionFeed()` を再実行
  - `gamePlaceholder`: `pendingChoices` があれば選択肢を再表示

## 10-bis. タイトル画面・エンドロールのブートシーケンス仕様

### タイトル画面（`#screen-title` / `title-screen.js`）

タイトル画面は `buzzutter_v2.html` のロード後に表示される初期画面。

| 要素 | 仕様 |
|---|---|
| `#screen-title` | `active` クラスで表示。ゲーム開始後は `classList.remove('active')` |
| BGM | `#titleBgm` を `_playTitleBgm()` で再生。ゲーム開始時に `_stopTitleBgm()` で停止 |
| デザイン切替 | `data-design="new"/"old"` で2種類のデザインを切替可能（`localStorage` で保持） |
| タイムライン演出 | `"new"` デザイン時に `_startTsTimeline()` / `_startTsTyper()` が発火しSNSタイムラインがアニメ表示 |
| 「はじめる」 | `startGame(false)` を呼び出しメインモード開始（`IS_DEBUG_MODE = false`） |
| 「続きから」 | `showLoadSlotPicker()` を呼び出しセーブスロット選択画面を表示 |

### ゲーム起動ブートシーケンス（`startGame()` / `ui.js`）

```
startGame(isDebugMode)
  ↓
_stopTitleBgm()
  ↓
#boot-overlay を active にしてブートアニメーション（3秒）
  ↓ 3000ms後
isDebugMode=true: #screen-charselect → buildCharSelect()（デバッグモード・ルート選択）
isDebugMode=false: startMainModeAutoRoute()（メインモード・みどり→朔→誠司→花蓮の順）
  ↓
overlay.fading → overlay 非表示
```

### メインモード自動進行（`startMainModeAutoRoute()`）

- `MAIN_MODE_ROUTES = ['midori', 'saku', 'seiji', 'karen']` の順でルートを自動進行
- `MAIN_MODE_ROUTE_INDEX` でどのルートか管理
- 1ルート終了（`retryGame()` 呼び出し）ごとに `MAIN_MODE_ROUTE_INDEX++`
- 全4ルート完了後（index >= routes.length）に `showEndroll(goToTitle)` を呼び出す

### エンドロール（`#screen-endroll` / `showEndroll()`）

全ルートクリア後、または各ルートのリトライボタン押下後に表示される。

| 条件 | 動作 |
|---|---|
| 全4ルート完了時（メインモード） | `showEndroll(goToTitle)` → エンドロール → タイトル画面へ |
| デバッグモードのリトライ時 | `showEndroll(() => buildCharSelect())` → エンドロール → キャラ選択へ |

```
showEndroll(callback)
  ↓
#screen-endroll を active にしてクレジットスクロールアニメ開始
  ↓
クリックまたはアニメ終了（animationend）で onEnd 発火
  ↓
#screen-endroll 非表示 → callback() 実行
```

### 未対応の潜在課題（ソースコードから確認）

1. **朔ルートのエンディング分岐ロジック**: `resolveEnding()` の saku ルートで `kShinonoya` が false の場合もデフォルト `'zure'` を返す実装になっており、到達エンド以外は必ずズレエンドになる。「自立エンドへの到達ルート不在」の設計上の意図確認が必要。

2. **キャラクター名の不一致**: `CHARACTERS` オブジェクト（app-core.js）の名前（「倉田 朔」「松本 誠司」）とマスター仕様書（「水無月 朔」「天羽 誠司」）で名前が異なる。正式名称を統一する必要がある。

3. **花蓮ルートの既読エンド達成難度**: `resolveKarenEnding()` は buzz/self のみで判定し kidoku を返す。`endings.js` の `condition` フィールドの `selfScore >= 65 かつ flashbackPhase >= 3` は **データ注釈**であり `showEnding()` 実行時には評価されない。buzz < 65 かつ self >= 50 の条件さえ満たせば kidoku エンドに到達できる（§5-4 参照）。ゲームデザイン上の「望ましい到達条件」と実装上の「到達可能条件」が乖離しているため、今後の仕様見直し時に実装強制を加えるか現状の設計を明示的に維持するか判断が必要。

4. **flashbackPhase の朔ルートでの累積**: Phase 2 は朔ルートでのみ発火するため、朔ルートをプレイしていない場合は花蓮ルートでも Phase 1 → Phase 3 のスキップが発生する可能性がある。全ルートを通じた整合性確認が必要。
