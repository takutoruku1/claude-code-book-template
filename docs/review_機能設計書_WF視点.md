# 機能設計書 レビュー（ワイヤーフレーム視点）

**レビュー実施日**: 2026-05-10  
**レビュアー**: UIワイヤーフレーム作成担当エージェント  
**対象バージョン**: 機能設計書 v2.0

---

## 総合評価

**B+（概ね良好・一部重大な記述漏れあり）**

機能設計書の全体的な記述密度は高く、GSオブジェクト・エンディング定義・スコア計算式など実装直結の情報は充実している。一方でUIワイヤーフレーム視点から見ると、「UIフロー上でどの画面がどのタイミングで切り替わるか」「mysteryCluesがUI上にどう見えるか」「素材カードのロック/アンロック状態の挙動」が設計書本文に記述されておらず、初めての開発者がワイヤーフレームなしでUIを実装することは困難な箇所が残る。

---

## 重大な問題点（修正必須）

1. **画面遷移トリガーの記述が欠落している**  
   各フェーズ（チャット → 素材確認 → 投稿ビルダー → 反響 → エンディング）間の「何を起点に画面が切り替わるか」が設計書本文に記載されていない。`trigger: showMaterial` や `trigger: showPost` という概念はシナリオフロー図（scenario_flow.html）には登場するが、機能設計書の §5「ルート機能詳細」にはトリガー名が一部ルートにしか明示されていない。具体的には §5-1（みどりルート）のチャットフロー表にトリガー行がなく、みどりルートではチャット終了後にどのタイミングで `.material-area` が表示されるかが設計書本文だけでは読み取れない。

2. **mysteryCluesのUI上の可視化方法が記述されていない（§7の不足）**  
   §7「ミステリーシステム」はmysteryCluesの蓄積ロジック（JS実装レベル）の記述に終始しており、「プレイヤーが画面上でどのようにミステリーの蓄積を確認できるか」が一切記載されていない。ワイヤーフレームでは主人公ウィジェットのモノローグバブルやシステムメッセージでのヒント表示を想定していたが、設計書上はUIへの反映仕様が空白になっている。花蓮ルートで `mysteryClues.length >= 2` の条件を満たすには「素材をめくってmysteryFlagを取得した」というフィードバックをプレイヤーが受け取る必要があるが、その表示方法（トースト通知か、カードタグか、メモ帳への記録か）が未定義のまま。

3. **素材カードのロック状態（locked状態）の開放条件が記述されていない**  
   UI設計書 §4-4では `.card-flip-wrapper.locked`（`opacity:.45; pointer-events:none`）の状態が存在することが明示されているが、機能設計書 §9「投稿ビルダー機能」の「素材選択ロジック」では「めくった後に再クリックで選択（最大2枚）」という手順のみが書かれており、カード自体がいつ「ロック」されるのか、あるいは全カードが最初から操作可能なのかが記述されていない。実装者がUI設計書のlockedクラスを見て混乱する可能性がある。

4. **花蓮ルートの既読エンド達成条件が設計書内で矛盾している**  
   §5-4では「既読エンド: buzz < 65 または self >= 50 の場合のデフォルト（resolveKarenEnding()による）」と記述されているが、§8「エンディング定義」では「selfScore >= 65 かつ flashbackPhase >= 3」が必須条件として記載されている。resolveKarenEnding()はbuzz/selfのみで判定し `kidoku` を返すが、showEnding()内の表示条件は別途selfとflashbackPhaseを要求するという2段構造が設計書の読み手に伝わっていない。§11「今後の実装課題」に指摘があるものの、設計書本文の条件表が修正されていないため矛盾が解消されていない。

5. **エンドロール・タイトル画面との接続仕様が機能設計書に存在しない**  
   全ルートクリア後に表示される `#screen-endroll`（エンドロール画面）と、ゲーム開始時の `#screen-title`（タイトル画面）の動作仕様について機能設計書に記述がない。ファイル構成表（§2）に `title-screen.js` は登場するが、「どの条件でエンドロールが発火するか」「タイトル画面からゲームを開始する際のブートシーケンス（`#boot-overlay`）の仕様」が設計書の目次に含まれておらず、UI設計書 §3との連携が断絶している。

## 改善推奨事項

1. **各フェーズの「入口条件」と「出口条件」を設計書に追記すべき**  
   現在のルート詳細（§5）は「選択肢ボーナス」「素材リスト」「エンディング条件」が中心で、フェーズ間遷移の条件（例:「clippableキーワードをメモしなくても素材フェーズには進める」「素材カードを1枚も選択せずに投稿フェーズへ進めるか」）が記述されていない。ワイヤーフレームの `.material-proceed-btn`（次へ進むボタン）の活性化条件と設計書の記述を一致させるため、各フェーズの進行条件を明示する節を追加すべき。

2. **投稿ビルダーの「投稿ボタン活性化条件」のUI表現を追記すべき**  
   §9では `selectedStyle != null && selectedHash != null && selectedTime != null` の全て揃った時に投稿ボタンが活性化すると記述されているが、「素材カードを1枚も選択していない状態でも文体・ハッシュ・時間を選べば投稿できるか」が不明確。ワイヤーフレームの `.post-opt.disabled` 状態の表示仕様と連動させるため、素材選択が投稿ボタン活性化の必須条件かどうかを明示すべき。

3. **チャットクイズ（CHAT_QUIZ）の仕様が設計書に記述されていない**  
   全ルートのシナリオフロー図にはクイズステップが明示されており（「クイズ: みどりさんが一番求めていることは」等）、正解・不正解で異なる分岐が存在する可能性を示唆している。しかし機能設計書のどのセクションにもクイズ仕様（出題タイミング・正解ボーナス有無・不正解ペナルティ・スキップ可否）が記述されていない。UIワイヤーフレームでは「クイズ画面はチャット選択肢UIを流用する」（UI設計書 §4-3）とされているが、スコアへの影響が不明なまま実装することになる。

4. **朔ルートのエンディング分岐ロジックの「意図的な仕様か否か」を明確化すべき**  
   §11で「kShinonoya が false の場合もデフォルト 'zure' を返す」という実装上の課題が指摘されているが、これが設計上の意図（朔ルートで到達エンド以外は必ずズレエンドになる）なのか、実装バグなのかが設計書本文の §5-2 エンディング条件表からは判断できない。条件表に「上記以外 → 到達エンド不達のためズレエンドに吸収（意図的）」のような注記を追加すべき。

5. **キャラクター名の正式名称を設計書内で統一すべき**  
   §4の表では「倉田 朔」「松本 誠司」、§11では不一致が指摘されているが、UI設計書 §6では「水無月 朔」「天羽 誠司」が使用されている。設計書内の注記（§4の脚注）だけでは不十分で、ゲーム内表示名・ルートID・CSS変数名・設計書内の名称を一覧化したマッピング表を追加すべき。

6. **セーブ/ロード時の画面復元とUIの対応を詳述すべき**  
   §10では `currentArea`（'gamePlaceholder' | 'materialArea' | 'postArea' | 'reactionArea'）に応じた復元処理が記述されているが、「素材カードのめくり状態・選択状態がどのように視覚的に復元されるか」のUI仕様（flippedCards・selectedCardsのDOM再適用とクラス付与の対応）が設計書から読み取れない。実装者が参照するとDOM復元の詳細が分からない。

## 軽微な指摘

1. **§9「投稿ビルダー機能」の素材選択ロジック第5項について**  
   「チャット中の画像も選択可能（selectedImageSrc）」と記述されているが、これがUI上でどのように表示されるか（素材カードと同一UIか、別エリアか）が不明。ワイヤーフレームの素材選択セクションとの対応が取れていない。

2. **flashbackPhase の段階1のトリガーが設計書と実装で異なる可能性**  
   §7ではPhase 1を「ゲーム開始直後（プロローグ）」としているが、UI設計書 §5-7（フラッシュバック演出）では段階1を「みどりルートの『お母さん』発言」としており記述が異なる。どちらが正しいかを設計書で明確化すべき。

3. **egoScoreの初期値と変動ルールの記述が不完全**  
   §3のGSオブジェクト定義では `egoScore: 0` と初期値のみ記述されているが、加算タイミングの一覧が §6「スコアシステム」に含まれていない。§5の各ルートには `egoPlus` の記述が散在しているが、統合的なegoScoreの変動ルール表がない。

4. **§6の「エンディング判定条件」コードブロックとズレエンドの優先判定順序**  
   全ルートでズレエンドが先に判定されるという実装上の動作は §5 各ルートのエンディング表に注記されているが、§6の `resolveEnding` コードブロック内のみどりルート疑似コードでは判定順序が `jiritu → zure → jizoku` の順に見えるが、実際は朔ルートと同様に `zure` が優先される可能性があり、読者に混乱を与える恐れがある。

5. **§2「モジュール間の依存関係」に ui.js・protagonist.js が含まれていない**  
   ファイル構成表（§2 上部）には `ui.js` と `protagonist.js` がリストアップされているが、モジュール依存関係図にはこれらが含まれておらず、UIコンポーネント側から game-logic.js を呼び出す逆方向の依存（例: `protagonistWidget` が `GS.route` を参照する）が記述されていない。

## 良かった点

1. **GSオブジェクト全変数の型・初期値・用途が一覧化されている**  
   §3のGSオブジェクト定義はコメント付きのJS形式で記述されており、変数の型・初期値・更新タイミング・用途がすべて1箇所で確認できる。UI実装時にどのステートを参照してUIを更新すべきかが一目で分かる構造になっている。

2. **素材カードの mysteryFlag・styleUnlock・hashUnlock が表形式で整理されている**  
   §5の各ルート素材リスト表は、カードIDとUIに直接影響する3つのプロパティが横並びで確認できるため、実装時に「このカードをめくったらどのボタンが解放されるか」をルックアップするのに適した形式になっている。

3. **エンディング8種の演出詳細が充実している**  
   §8では各エンディングのBGMトラック・フェードアウト秒数・暗転秒数・monologueInterval・clientMessage・特殊演出フラグが全て列挙されており、エンディング画面（`#screen-ending`）の実装仕様として直接参照できるレベルで記述されている。

4. **スコア計算式が擬似コードで明示されている**  
   §6のスコア計算式（chatBuzzBase + styleOptions[i].buzz + hashOption[j].buzz）と時間帯ボーナス表が数値ベースで記述されており、UIのスコアバー（`.score-fill`）の幅計算に直接使える形になっている。

5. **「今後の実装課題」セクションで既知の問題点が誠実に記録されている**  
   §11では実装済み修正と未対応課題が明確に分けて記述されており、設計書が「現状の実装を正確に反映したもの」として機能していることが確認できる。特に花蓮ルートの既読エンド達成難度やflashbackPhaseの累積問題など、テスト時に問題になりうる箇所が事前に文書化されている点は評価できる。

6. **セーブデータのJSONスキーマが詳細に記述されている**  
   §10のセーブデータ構造はキー名・型・用途が全て記述されており、ウィンドウ状態のフィールド（appWindowActive・appMinimized等）まで含めた完全なスキーマ定義になっている。セーブ/ロードのUIを実装する際の参照資料として十分な品質がある。
