# UIワイヤーフレーム レビュー（機能設計視点）

レビュー実施日: 2026-05-10  
レビュー対象: docs/UIワイヤーフレーム.html  
参照資料: docs/機能設計書.md / js/scenario.js / js/endings.js / js/postOptions.js

---

## 総合評価

**B（概ね良好・重大な欠落あり）**

7つの画面フェーズはすべてタブとして網羅されており、スコアバーやエンディング切り替えデモなど実用的な確認機能が揃っている。ただし、機能設計書が規定する「8種エンディング」に対しワイヤーフレームが「7種」しか提示していない欠落、投稿ビルダーの機能仕様との複数の不整合、および時間帯選択・素材ロックアウト機能の完全欠如など、実装者が誤解を生じる箇所が複数存在する。

---

## 重大な問題点（修正必須）

1. **継続エンド（jizoku）の欠落**  
   機能設計書 §8 は全8種エンディングを定義している（jiritu / jizoku / toutatu / saihan / kidoku / zure / zange / chinmoku）。ワイヤーフレームのエンディング画面（PANEL 6）に用意されているタブは7種（自立・到達・再訪・既読・ズレ・贖罪・沈黙）であり、みどりルートのデフォルトエンドである「継続エンド（jizoku / 🌱）」が完全に抜け落ちている。endings.js に `jizoku` の定義（🌱・「大きくなくていい、と」）は実装済みであり、ワイヤーフレームに追加タブと対応するモックが必要。

2. **投稿ビルダーに時間帯選択が存在しない**  
   機能設計書 §6 は投稿ビルダーのスコア計算に「時間帯ボーナス（morning/noon/evening）」を規定しており、STEP 3 相当として `selectedTime` を確定する操作が必要とされる。ワイヤーフレームのSTEP 1〜3（文体・素材・ハッシュタグ）にはこのUIが存在しない。投稿ボタンの活性化条件（`selectedStyle != null && selectedHash != null && selectedTime != null`）も画面上に注記がない。実装者がこのSTEPを実装し忘れるリスクが高い。

3. **文体オプションのラベルが機能仕様と不一致**  
   機能設計書 §9 および postOptions.js の `styleOptions` は以下4種を定義している：
   - 0: 短くシンプルに（buzz:30, self:80）
   - 1: 丁寧に感情を込めて（buzz:50, self:70）
   - 2: バズを狙って大胆に（buzz:85, self:15）
   - 3: 失敗も含めてリアルに（buzz:45, self:90）
   
   ワイヤーフレームの文体ボタンは「日常的な語り口」「感情を前面に」「バズ狙いの断言型」「静かな等身大」という異なるラベルになっており、スコア注記（「egoScore +1」「selfScore ↑↑」）も実際の仕様と合っていない。実装仕様ドキュメントとして参照される場合、実装者が誤ったラベルや値で実装する恐れがある。

4. **素材カードのロックアウト機能（styleUnlock/hashUnlock）が表現されていない**  
   機能設計書 §9 は「解放されていない選択肢ボタンは `disabled` 状態」と規定しており、選択した素材カードによって文体・ハッシュタグの選択可能範囲が変化する。ワイヤーフレームの投稿ビルダーではすべての文体・ハッシュタグが同時に活性化しており、このロックアウト機構が一切表現されていない。UI上で「選択不可状態」のボタンを1例でも示す必要がある。

## 改善推奨事項

5. **ハッシュタグがみどりルート固定で他ルートを表現していない**  
   postOptions.js の `HASH_OPTIONS_BY_ROUTE` はルート別に異なるハッシュタグを定義している（みどり: #植物のある暮らし、朔: #ジオラマ #模型制作、誠司: #喫茶店 #純喫茶、花蓮: #生け花 #いけばな）。ワイヤーフレームは一律「#今日の一枚」「#植物のある暮らし」「#日常の記録」「#家族」「#完璧じゃなくていい」「#インスタ映え」と表示しており、キャラクター別の動的切り替えが示されていない。少なくとも「ルートにより変化」の注記と別ルートの例を1件追加すること。

6. **egoScoreの変動ロジックがワイヤーフレームに説明されていない**  
   スコア表示画面（PANEL 5: 反響フェーズ）では selfScore と buzzScore の2値のみバー表示されているが、egoScore はエンディング画面（PANEL 6）で突然数値として出現する。機能設計書 §3 は egoScore の変動タイミング（選択肢でegoPlus/egoPlusが発生する）を規定しており、チャット画面やエンディングへの到達ロジックに関わる。egoScore の加算タイミングをチャット画面またはエンディング画面の仕様注記に追加すること。

7. **素材詳細モーダルが「未実装のため省略」とされているが仕様が不明確**  
   PANEL 3（素材確認画面）の注記に「クリックで素材詳細モーダル展開（未実装画面のため省略）」と記載されている。機能設計書 §9 は「カードをめくって裏面を確認」→「再クリックで選択」の2段階操作を規定しており、「めくり（裏面表示）」と「選択（selectedCards への追加）」が視覚的に別操作であることをワイヤーフレームが示していない。最低限、カードの「めくり後状態」と「選択済み状態」の見た目の違いを追加すること。

8. **エンディングの分岐条件注記に誤りがある（kidokuの条件）**  
   PANEL 6 の annot-block に記載されたエンディング分岐条件：  
   「🌸 既読エンド: mysteryClues ≥ 2 + k_shinonoya（花蓮ルート）」  
   しかし機能設計書 §5-4 および endings.js の定義によると、既読エンドの実際の条件は「selfScore >= 65 かつ flashbackPhase >= 3（花蓮ルート）」であり、mysteryClues は「謎収束選択肢の出現条件」であって「既読エンド自体の到達条件」ではない。k_shinonoya は到達エンド（朔ルート）の条件フラグであり、既読エンドには無関係。記述を修正すること。

9. **フラッシュバックのフェーズ仕様が不完全**  
   PANEL 1（チャット画面）の注記は「朔ルート Stage 1〜3 でトリガー」と記述されているが、機能設計書 §7 はフラッシュバックのフェーズを「Phase 1（ゲーム開始直後プロローグ）」「Phase 2（朔ルート saku_fb2 trigger）」「Phase 3（花蓮ルート k_memo カードめくり時）」と定義している。Phase 1（プロローグ）は朔ルートではなくゲーム開始直後であり、Phase 3 は花蓮ルートであるため説明が不正確。また Phase 3 の「現在カードと過去記憶の2枚並列表示」という特殊演出がワイヤーフレームに全く示されていない。

10. **反響フェーズのエンド条件注記が一部のみ**  
    PANEL 5 のスコアボックス下部注記「エンド条件: self ≥ 65 → 自立エンド | buzz ≥ 65 + self < 50 → ズレエンド」はみどりルートの条件のみを示している。朔（kShinonoyaフラグ必要）・誠司（デフォルト再訪）・花蓮（resolveKarenEnding）ではルート別に異なる条件が適用される。注記に「ルートによって条件が異なる（詳細は機能設計書 §6 参照）」の一文を追加することを推奨。

## 軽微な指摘

11. **クイズのプログレスバーが5ドット固定だが、みどりルートは2問**  
    PANEL 2（クイズ画面）のプログレスバーは常に5ドット表示されている。機能設計書 §5（クイズ注記：みどり：2問、朔/誠司：3問、花蓮：4問）に基づけば、ドット数はルートにより変化する。ワイヤーフレームはデモ用固定値として許容範囲だが、「問題数はルートにより変化」の注記があると実装時に混乱が生じない。

12. **チャット画面のフレンドリスト（talksList）に誠司・花蓮が表示されていない**  
    PANEL 1 のフレンドリスト（chatTalksList）にはみどり🌿と朔🌙のみが表示されている。機能設計書 §4 では4人のクライアント（みどり / 朔 / 誠司 / 花蓮）が定義されており、全クライアントが一覧に並ぶUI構造がワイヤーフレームに反映されていない。実際のゲームではメインモードで4ルートを順に進む構成であることを考えると、フレンドリストの全件表示モックが望ましい。

13. **朔のアバター絵文字が機能設計書と異なる**  
    PANEL 1 のフレンドリストで朔のアバターは「🌙」で表示されているが、機能設計書 §4 の CHARACTERS 定義では朔のアバターは「🔧」とされている。タイトル画面の浮遊カードでも「@saku_craft」で「🌙」が使われており、機能設計書の「@saku_waterless」とも不一致がある。設計資料として統一が必要。

14. **エンドロール画面がクレジット内容のみで、機能仕様が示されていない**  
    PANEL 6 下部にエンドロール参考UIがあるが、「クリックでスキップ」以外の仕様（スクロールアニメーション速度、BGM連動、タイトルへの自動遷移タイミングなど）が全く記述されていない。エンドロールは `showEndroll()` 関数で制御される独立した演出であるため、スペック注記の追加を推奨。

15. **セーブ/ロード機能を表す画面が存在しない**  
    機能設計書 §10 は3スロットセーブ・ゲーム中ロード・タイトルからロードを詳細に規定しているが、ワイヤーフレームにこれらのUI（セーブスロット選択ピッカー・ロードメニューなど）が一切含まれていない。全7画面の範囲外として意図的に省略されているなら「セーブ/ロードUIは別途ワイヤーフレームを作成予定」等の注記が必要。

## 良かった点

1. **全7フェーズを網羅したタブ構成**  
   タイトル→チャット→クイズ→素材確認→投稿ビルダー→反響フェーズ→エンディングという正しいゲームフロー順にタブが並んでおり、SCREEN 01/07 の番号付けも一貫している。

2. **フラッシュバック演出の視覚的表現**  
   PANEL 1 でフラッシュバックバー（FLASHBACK / BACK TO PRESENT）と黒背景斜体バブルを用いた演出が具体的に示されており、実装者が直感的に理解できるモックになっている。

3. **mysteryClueカードの視覚的区別**  
   PANEL 3 でミステリーカード（パープル枠・🔍 MYSTERY バッジ・「謎のヒントが含まれています」）が通常カードと明確に区別されており、機能設計書の mysteryFlag の視覚化として適切。

4. **egoScoreを含む3スコア全ての表示**  
   エンディング画面（PANEL 6）でself score / buzz score / ego scoreの3値を並列表示しており、機能設計書 §3 の GS オブジェクト設計と整合している。

5. **エンディングデモのインタラクティブ切り替え**  
   7種エンディング（継続エンド欠落を除く）のタブ切り替えデモが動的に動作し、各エンディングのグローカラー・バーカラー・テキストの差異が確認できる。設計レビューツールとして実用性が高い。

6. **コメントフィードのselfScore/buzzScore連動説明**  
   PANEL 5 の注記に「selfScore が高いと共感系、buzzScore が高いとスパム系コメントが増える演出」と記述されており、endings.js / scenario.js のリアクションキー判定（high_self / high_buzz / balanced）の視覚的意図が説明されている。

7. **投稿プレビューのリアルタイム更新の概念**  
   PANEL 4 右側に「PREVIEW」エリアとスコア予測パネルを並置し、選択によってスコアが変化するインタラクションコンセプトが明示されている。
