## 総合評価: B

演出命令セットの設計精度と実装カバレッジは高く、基幹システムとして機能する水準に達している。一方で、シナリオ・世界観の視点からは「演出の強度設計がセリフの感情強度と乖離している箇所」「フラッシュバックのセリフが設計書の正規版と異なる」「『翻訳』テーマが演出から欠落している」の3点が看過できない問題として残る。修正コスト自体は高くなく、指摘箇所を対処すれば A 評価が射程に入る。

## 重大な問題点

### 問題①: フラッシュバック Phase 2 のモノローグが設計書と齟齬している

**該当箇所**: `direction_script.md` §4-2「Phase 2」

演出スクリプトに記述されたモノローグ:
```
{ cmd: 'mono', text: '——残す、か。', style: 'flashback', durationMs: 2000, force: true },
{ cmd: 'wait', ms: 1800 },
{ cmd: 'mono', text: '証拠を残す。それが俺の仕事だった。', style: 'flashback', durationMs: 2400, force: true },
```

`plot_steinsgate.md` で確定しているテキスト（朔ルート・フラッシュバック強化版）:
```
「——残す。あの記事も、残っている。」
「2200文字。送信まで8分かかった。」
「@k_shinonoya のアカウントが削除されたのを確認したのは、翌朝の5時だった。」
「コメント欄は302件。私が書いたのはその火種の一つだった。」
```

**何が問題か**: 演出スクリプトのテキストは「仕事への向き合い方」を語る中立的な回想に見える。設計書のテキストは「数字の列挙（2200文字・8分・翌朝5時・302件）」によって加害行為の具体性を刻みこむ設計だ。「証拠を残す。それが俺の仕事だった。」という一文は「仕事として正当化していた」という告白に読めてしまうが、設計書で意図しているのは「加害の証拠が今もある」という自己糾弾だ。キャラクターの倫理的立場が演出テキストの差異によって別人になっている。

**修正方向**: Phase 2 の `mono` テキストを `plot_steinsgate.md` §朔ルート「追加フラッシュバック強化テキスト」の全文と置換する。4行を複数の `mono` + `wait` のセットに分割して実装する。

---

### 問題②: フラッシュバック Phase 3 のオーバーレイテキストが「観察者の語り」になっている

**該当箇所**: `direction_script.md` §2-4「k_memoめくり時の演出」

演出スクリプトのオーバーレイテキスト:
```
text: '3年前の記事。\n篠宮カレンの姉。\n俺が——追っていた。'
```

その後の `mono`:
```
text: '——全部、繋がっていた。'
```

`plot_steinsgate.md` §花蓮ルート「湊が気づく瞬間の設計」で確定しているモノローグ:
```
「——篠宮。」
「カレンが好きそう、という名前。ゆきわりそう。既読のつかないメッセージ。」
「頭の中で、点と点が線になっていく。」
「線になった瞬間——吐き気がした。」
```

**何が問題か**: 演出スクリプトのオーバーレイテキストは「3年前の記事 / 篠宮カレンの姉 / 俺が追っていた」という3項目を列挙する「整理された要約」になっている。これは「気づいた後」の反芻であって、気づく瞬間の身体的な衝撃（「吐き気がした」）ではない。設計書のクライマックスは「吐き気」という身体感覚で加害の重さを表現する構造だが、演出スクリプトでは知的な整理として描かれており、感情の核が抜け落ちている。

また `overlay_text` で「3年前の記事。篠宮カレンの姉。俺が——追っていた。」という構造化された文章を中央大書きにする演出は、ノワール的なかっこよさが出てしまうリスクがある。このシーンは湊が英雄的に「真実を掴む」場面ではなく、自分が何をしたかを正面から思い知る場面である。

**修正方向**: `overlay_text` の使用を再検討する。代わりに `mono` の連打（`style: 'flashback'`、短い `wait` を挟む）で設計書のテキストを忠実に実装する。オーバーレイ形式が残るなら、テキストを「——吐き気がした。」の一行だけにして、そこに全重量を集中させる。

---

### 問題③: 「翻訳」テーマが演出設計から完全に欠落している

**該当箇所**: `direction_script.md` 全体

`plot_steinsgate.md` では「翻訳」という言葉がゲーム全体を貫く中心テーマとして設定されている。みどりルートで仕事への誇りとして登場し、花蓮ルートの贖罪エンドで「翻訳、ありがとうございました。」として収束する。この収束こそがシュタゲ的構造の核心だ。

演出スクリプトには「翻訳」に言及する演出が一切ない。特に以下の場面に演出が必要である:

1. **みどりルートの `midori_mother`**: 現状 `{ cmd: 'mono', text: '（…言葉が、ある）', style: 'normal' }` と記述されているが、この瞬間こそ「翻訳」というテーマが初めて肉体を持つ場面だ。設計書では「翻訳が上手くいった、と思う」という湊のモノローグが確定している。演出スクリプトが別テキストを指定していることで、実装時に矛盾が生じる可能性がある。

2. **贖罪エンド直前のモノローグ連鎖**: 設計書 §花蓮ルート「既読エンドのモノローグ」に詳細なテキストが設計されているが、演出スクリプトには `karen_zange_response` 以降の演出が心拍音と既読演出どまりで、エンディングへの橋渡しとなるモノローグ演出が設計されていない。

**修正方向**: みどりルートの `midori_mother` モノローグテキストを設計書の確定テキストに合わせる。贖罪エンド経路（`karen_zange_response` → エンディング）に、設計書確定の「翻訳者というのは消えることだと思う」モノローグを `mono` 連打で追加する。

## 改善推奨

### 推奨①: 朔ルートの「DM通知」モノローグのニュアンス調整

**該当箇所**: §2-2「saku_dilemma2 ステップ」

```
{ cmd: 'mono', text: '（……聞いたことのある、アカウント名だ）', style: 'hollow', durationMs: 2800 }
```

`plot_steinsgate.md` 確定テキスト:
```
「k_shinonoya——」
「3年前に削除されたはずのアカウント名と、最初の4文字が同じだった。」
「偶然だ、と言い聞かせた。」
```

演出スクリプトの「聞いたことのある、アカウント名だ」は印象が薄い。「知っているかもしれない」という曖昧さを出す意図は分かるが、「3年前に削除されたはず」「偶然だ、と言い聞かせた」という設計書のテキストの方が、抑圧と気づきの拮抗が強く出る。実装時には設計書テキストを優先すること。1つの `mono` に全文を詰め込むより、`wait` で区切って3行を別々の `mono` で送る形を推奨する。

---

### 推奨②: `saku_dm_name_reply`「嘘をついた」演出の強化

**該当箇所**: §2-2「saku_dm_name_reply ステップ」

```
{ cmd: 'mono', text: '（嘘をついた）', style: 'hollow', durationMs: 2200 }
```

3文字の `mono` を 2200ms 表示するのは正しい判断だ。ただし「嘘をついた」の前に `wait: 1200` が入っている現状では、プレイヤーが返答文を読んでから 1.2 秒後にモノローグが出る計算になる。「……いえ。他人です。気にしないでください。」という返答テキストを読了するタイミングと、モノローグ表示のタイミングを合わせるために `wait` を 600ms 程度に短縮することを推奨する。感情の鮮度が最も高い瞬間に「嘘をついた」を刺す設計の方がキャラクターとして誠実だ。

---

### 推奨③: 花蓮ルートの `karen_mystery_choice` 直前に花蓮の問いかけシーンを接続する

**該当箇所**: §2-4「karen_mystery_choice ステップ演出」

演出スクリプトの `karen_mystery_choice` は、`bgm_change: silence` から始まってオーバーレイ・モノローグまで完結した演出シーケンスとして設計されている。しかし `plot_steinsgate.md` §花蓮ルート「花蓮が語る場面」では、`karen_mystery_choice` の直前に花蓮が「ひとつだけ、聞いてもいいですか。」と問いかけ、「3年前、SNSで誰かに書かれた記事で、姉が消えました。」と告白する会話シーンが確定している。

この会話シーンと `karen_mystery_choice` の演出シーケンスの間に、演出の「橋渡し」が必要だ。花蓮が「湊さんが、誰なのか——私は最初から、知っていました。」「だから、頼んだんです。」と言った後の 3秒の沈黙に、現在の演出スクリプトは対応していない。この沈黙（花蓮の最後の言葉から `karen_mystery_choice` 選択肢表示まで）に `silence_input + wait + bgm_change: silence` を組み合わせた演出を追加することを推奨する。

---

### 推奨④: mystery フェーズ自動モノローグのテキストの再検討

**該当箇所**: §3-3「`_onMysteryPhaseUp()` 関数」

```javascript
const monoTexts = {
  1: '（……何か、引っかかる）',
  2: '（同じ場所が、また出てきた）',
  3: '（——全部、繋がっている）',
};
```

フェーズ3のモノローグ「（——全部、繋がっている）」は、花蓮ルートのクライマックス演出（§2-4）でも使用されている「——全部、繋がっていた。」に酷似している。フェーズ3への移行が花蓮ルートのクライマックス前に起きた場合、クライマックスの言葉の重量が先取りされて消費される恐れがある。フェーズ3の自動モノローグは「（——止まれない）」や「（考えるな）」など、気づきを抑圧しようとする内面を示すテキストへの変更を推奨する。

## 良かった点

### 良①: `silence_input` 演出の「既読」演出としての完成度

`directionSilenceInput()` の実装および `kidoku-overlay` のHTML/CSS設計は、このゲームが SNS の「既読」という記号に込めた意味を的確に体現している。チェックマーク2重表示・フェイク入力欄・カーソル点滅という3要素の組み合わせは、「送れない」ではなく「送ったが届かない（または届いた後に何もできない）」という状態をUIとして正確に表現している。「既読」テーマは `plot_steinsgate.md` の既読エンドの根幹であり、その演出を単なる「入力欄の無効化」に終わらせず、視覚的語りにした判断を評価する。

---

### 良②: `showUnsentMessage()` 関数の「送れないメッセージ」演出

```javascript
function showUnsentMessage(ghostText, onEnd) {
  // タイプ → 2秒後に全選択 → 削除
}
```

花蓮ルートの `karen_zange_response` 後に「……あなたは、」「ごめん、」と入力されかけて消えるという演出の設計は、シナリオ設計書が求める「言葉が出てこない湊」を実装レベルで体現している。特にテキストが「ハイライト（選択状態）」になってから消えるという表現は、「書こうとしたが選択肢を失った」という行為の可視化として優れている。セリフに依存せず行動で感情を示す演出として、世界観と整合している。

---

### 良③: みどりルートの演出を最小化する判断

§2-1「みどりルートは演出なし（チュートリアル）」として `direction` を極力排除した設計は正しい。みどりルートは感情的負荷を意図的に低く保つことで、後続ルートとのコントラストを強化する設計だ（`plot_steinsgate.md` §4「感情弧マップ」参照）。演出過多になれば「チュートリアル感」が消え、後半ルートの演出強度が相対的に低下する。「演出しないことで語る」という判断が正しく適用されている。

---

### 良④: BGMトラック名のセマンティクス設計

`tension_low` / `tension_high` / `silence` / `ending_piano` / `ending_strings` という5トラック構成は、感情弧の段階と一致している。特に「花蓮クライマックスで `silence` に切り替える」という演出選択は、BGMを消すことで「沈黙そのものが演出になる」という正しいアプローチだ。BGMがないことで花蓮の「知っていました。」というセリフが空白に落ちる設計は、シナリオが狙う「言葉の重さ」を音響的に補強している。

---

## まとめ

演出命令システムの設計品質は高く、概念設計としての完成度は A 水準に近い。しかし実装時に参照すべきセリフが `plot_steinsgate.md` の確定テキストではなく草稿レベルの別バージョンで記述されている箇所が複数あり、このまま `scenario.js` へ実装されると世界観とキャラクター描写に齟齬が生じる。フラッシュバック Phase 2・Phase 3 のセリフテキスト、および「翻訳」テーマの演出欠落という3点を重大問題として対処を求める。修正後に再レビューを推奨する。
