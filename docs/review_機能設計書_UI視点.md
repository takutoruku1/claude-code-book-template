# 機能設計書 レビュー（UI設計視点）

**レビュー日**: 2026-05-10  
**レビュー対象**: docs/機能設計書.md (v2.0)  
**レビュー実施者**: UI設計担当（UI設計書作成視点）  
**照合ソース**: js/scenario.js / js/game-logic.js / js/endings.js

## 総合評価

**B（概ね良好・要修正点あり）**

ゲームステート・スコア計算・エンディング定義はコードと高い整合性を保ち、UI実装の基礎資料として十分な情報量がある。ただし、(1) いくつかの記述とソースコードの間に実際の誤り・矛盾があること、(2) UI実装時に必要な画面遷移の詳細や条件が省略されている箇所があること、が課題。特に花蓮ルートのエンディング到達条件の記述は誤解を招く可能性が高く、優先的な修正が必要。

---

## 重大な問題点（修正必須）

1. **花蓮ルートの既読エンド到達条件の記述が不正確**  
   - 設計書 §5-4「エンディング到達条件」の表に「buzz < 65 または self >= 50 の場合のデフォルト（`resolveKarenEnding()` による）」と記載されているが、実際の `resolveKarenEnding()` は buzz/self の条件のみを判定して `'kidoku'` キーを返す。`self >= 65` および `flashbackPhase >= 3` のチェックは **`showEnding()` 内には実装されておらず**、`endings.js` の `condition` フィールドはデータの注釈にすぎない（実行時に評価されない）。  
   - つまり「self が 65 未満でも kidoku エンドに到達できる」のが現実の実装であり、設計書の記述（「`resolveKarenEnding()` によるデフォルト」の後に `self >= 65 かつ flashbackPhase >= 3` が「必要」と書く）は矛盾している。  
   - UI実装担当は「既読エンドには数値条件がある」と誤解しうる。この条件がランタイムで強制されるのか、演出上の前提条件なのかを明記すること。

2. **朔ルートのエンディング分岐ロジックの誤記述（§5-2）**  
   - 設計書の表には「到達エンド: self >= 60 かつ kShinonoya = true」「ズレエンド: buzz >= 65 かつ self < 50（先に判定）」の2行のみが記載されている。しかしソースコードの `resolveEnding()` では、これら以外のケース（self < 60 かつ kShinonoya = false、またはズレ条件以外）も全て `'zure'` を返す。設計書の表には「それ以外 → ズレエンド」という行がなく、設計書上は「ズレエンドでも到達エンドでもない場合の挙動が不明」に読める。§6 のコードブロック「朔ルート」には正しく記載されているが、§5-2 の表と矛盾している。UI上で結末ボタンの表示条件やプレイヤーへのフィードバックを設計する際に混乱の原因となる。

3. **素材カードの「めくり」と「選択」の操作仕様が不明確**  
   - §9「投稿ビルダー機能」には「クリックしてめくる（裏面を確認）」「めくった後に再クリックで選択」と記載されているが、「めくっていないカードを直接選択できないこと」「2枚選択済み時に別カードをクリックしても無視されること（`return` で早期脱出）」「locked 状態の視覚表現」についての記述がない。実コードの `lockExcessCards()` / `handleCardClick()` にこのロジックがあるが、UI仕様書側からすると「disabled 状態」との違いが何かを判断できない。UI設計書との対応を明示すること。

4. **素材カードのカード枚数（ルート別）が記載されていない**  
   - みどりルート: 4枚、朔ルート: 4枚、誠司ルート: 3枚、花蓮ルート: 3枚と実際には異なるが、設計書には「最大4枚」と一律に記載されている。誠司・花蓮ルートの実装では 3枚しかなく、UI レイアウト設計（グリッド列数・カードサイズ）に影響する。

## 改善推奨事項

1. **`updatePreview()` の実行タイミングが投稿ビルダーの仕様から欠落している**  
   - 設計書 §9 は「`updatePreview()` は selectedStyle / selectedHash / selectedTime 全て揃った時のみ投稿ボタンを活性化する」と記載しているが、実際には `updatePreview()` は選択肢クリックのたびに常に呼ばれており、スコアバー（buzzBar / selfBar）のリアルタイム更新も行う。投稿ボタンの活性化条件だけでなく「スコアバーのリアルタイム更新のタイミング」としてUIイベント仕様に追記すること。

2. **時間帯ボタン（selectedTime）がスコアに影響するタイミングの記述が不明瞭**  
   - §6「スコア加算ルール」の時間帯ボーナス表は `submitPost()` 内で適用されると記載されているが、`updatePreview()` での事前表示（スコアバーへの反映）は時間帯ボーナスを含まない（`updatePreview()` はボーナス未適用の状態で計算される）。UI的には「選択前後でスコアバーの値が変わるのか、変わらないのか」が重要な設計情報であり、明確に分けて記述すること。

3. **反響フェーズのフォロワー増減表示の仕様が設計書に記載されていない**  
   - §6 のリアクションキー判定表には「いいね / RT / フォロワー増減」の数値が記載されているが、UI上での表示形式（`▲234` などの増減矢印表示・旧フォロワー数と新フォロワー数の表示）が設計書に存在しない。実コード `buildReactionFeed()` に `arrow` / `cls` / `newTotal` 等の実装があるため、UI設計書側の反響フェーズ仕様と突き合わせると不整合が出る可能性がある。

4. **贖罪エンド（zange）の発火ルートが設計書とソースで乖離している**  
   - 設計書 §8 の贖罪エンドには「`typingText: '篠宮さん。3年前、記事を書いたのは私です。'`」の演出が記載されているが、`scenario.js` の `karen_zange_response` ステップでは花蓮が「知っていました。」と答える処理がシナリオとして実装されており、これは `endings.js` の `karenResponse` フィールドの内容と同一テキストで二重に存在している。UI実装時にどちらのチャットシステムが担当するのかが不明。エンディング演出（`showEnding()` 経由）とシナリオチャット（`runChatId()` 経由）の担当分離を明記すること。

5. **flashbackPhase のUI表現仕様が設計書に記載されていない**  
   - §7 にフラッシュバックの発火条件・テキスト概要は記載されているが、各フェーズのUI表現（CRT風テキスト / 暗赤色吹き出し / 2枚並列表示の具体的なDOM構造・アニメーション仕様）が一切記載されていない。UI設計書側にこれらの演出仕様があるか確認し、ない場合は機能設計書に補足するか UI設計書への参照を追記すること。

6. **セーブデータの `currentStep` フィールドの値域が記載されていない**  
   - §10 の `currentStep: Number` について「現在のステップ番号（1〜6）」とだけ記載されており、各番号が何の状態に対応するかの対応表がない。UI側でプログレス表示・ロード時の復元画面選択に使う重要情報であるため、ステップ番号とゲーム状態の対応を明記すること（例: 1=チャット中、2=素材選択、3=投稿ビルダー、4=反響フェーズ、5=クライアント反応、6=エンディング）。

## 軽微な指摘

1. **§8 エンディング一覧表の「主な条件」列が実装と微妙に乖離している**  
   - `saihan`（再訪エンド）の条件として「buzzScore < 65 かつ selfScore < 65」と記載されているが、実際の `resolveEnding()` での誠司ルートは「buzz >= 65 かつ self < 50 でなければ saihan」であり、self の条件（< 50 vs < 65）が表と異なる。

2. **§9 styleOptions の加算値がソースと逆転している**  
   - 設計書の表で「短くシンプルに: buzz+30 / self+80」「丁寧に感情を込めて: buzz+50 / self+70」と記載されているが、ゲームコンセプトの「らしさ vs バズ」のトレードオフとして整合しているかを確認すること（self が buzz を大幅に上回る選択肢が2つある）。実コードの `styleOptions` と数値一致は確認済みだが、設計意図がわかりにくい。

3. **§3 GS 定義で `selectedTime` の型が `String | null` と記載されているが初期値の記述がない**  
   - 他フィールド（buzz, self, etc.）は `0` などの初期値が記載されているのに対し、`selectedTime` だけ初期値の明示がない。`null` が初期値であることを明記すること。

4. **§5-3 誠司ルートの「clippable キーワード」取得パスの記述が不正確**  
   - 「`seiji_send`: 映える店じゃない（chatSelfBonus +8）」とあるが、実際のシナリオでは `seiji_shop_reply` で「その『古さ』が強みかもしれません」を選んだ場合にのみ `seiji_send` ステップに到達してこのキーワードが発話される。buzz側の選択肢（`seiji_send_buzz`）からは `seiji_send` を通らずクリップ不可能。条件分岐の有無を注記すること。

5. **§7 ミステリーシステムで `mysteryClues.length >= 2` の達成可能性の説明が不足**  
   - 「朔ルートで `s_desk` と `s_model` の両方をめくる必要がある」と記載されているが、誠司ルートで `sj_sign` をめくれば `yukiwarisou` が追加されること（つまり朔+誠司のクロスルートでも達成可能か否か）の言及がない。マルチルートゲームとして、どのルート組み合わせで条件を満たせるかを網羅的に記述すること。

6. **§4 キャラクター定義の注記について**  
   - 「倉田 朔」vs「水無月 朔」の名前不一致を注記しているのは良いが、ゲーム内の表示（ビジュアル上・チャット名）として現在どちらが使われているかを明記すること。プレイヤーに見えるキャラクター名として何が正しいかをUI設計書では判断できない。

## 良かった点

1. **GS オブジェクトの全変数定義が型・初期値・用途コメント付きで網羅されている**  
   ソースコードの `initGS()` との照合で概ね一致しており、UI/ロジック間のデータ共有仕様として非常に使いやすい。

2. **スコア計算フローが段階別（チャット蓄積 → ベースコピー → 選択加算 → 時間帯加算）に整理されている**  
   `chatSelfBonus` → `chatSelfBase` → `GS.self` の流れが明確に記述されており、UIプレビュー実装時の計算モデルとして参照しやすい。

3. **エンディング定義（§8）が `endings.js` のフィールドと正確に対応している**  
   `fadeDuration` / `monologueInterval` / `bgmDelay` / `noBlackout` 等の演出パラメータが全エンドについて記載されており、UI演出実装の設計書として機能している。

4. **ミステリーシステムの蓄積タイミングが一覧表とコードスニペットで説明されている**  
   `handleCardClick()` のコードスニペットとルート別カードID対応表の併記により、実装者が動作を理解しやすい構成になっている。

5. **「今後の実装課題」セクション（§11）に未対応課題が明記されている**  
   既知のバグ・設計上の懸念点が残課題として整理されており、引き継ぎや追加開発時の参照に有用。特に花蓮ルートの既読エンド達成難度問題・flashbackPhase の累積問題は、UI設計にも影響する重要な指摘として評価できる。
