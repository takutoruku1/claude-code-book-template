# UI設計書 レビュー（機能設計視点）

> レビュー実施日: 2026-05-10  
> レビュー対象: `docs/UI設計書.md`（v1.0）  
> 参照資料: `docs/機能設計書.md`（v2.0）、`css/main.css`、`js/ui.js`

---

## 総合評価

**B（概ね良好だが、機能仕様との重大な乖離が複数存在する）**

UI設計書はデザイントークン・コンポーネント構造・アニメーション定義を丁寧にまとめており、ベースとしての品質は高い。ただし、機能設計書の核心である「egoScore の第3スコア軸」「8種エンディングのUI差分」「mysteryClues 蓄積の画面への反映」が UI 設計書では未記述または不十分であり、実装担当者がUI設計書だけを見ると仕様を誤解・見落とすリスクがある。

---

## 重大な問題点（修正必須）

1. **egoScore（第3スコア軸）のUI表現が完全に欠落している**  
   機能設計書では `GS.egoScore`（+1/-1 で変動する「業スコア」）が贖罪エンド・沈黙エンドの分岐条件として定義されている。UI設計書の「4-5. 投稿ビルダー画面」セクションのスコアバーは `buzz`（ピンク）と `self`（グリーン）の2本のみ記載しており、egoScore の視覚化方法（バー・アイコン・非表示など）が全く言及されていない。花蓮ルート固有のスコア軸であるため、少なくとも「花蓮ルート時の egoScore 表示方針（非表示・専用バー・サマリー画面のみ等）」を明記する必要がある。

2. **エンディング画面（4-7）の8種UI差分が記述されておらず、実装不能な状態**  
   機能設計書では各エンディングに固有の演出仕様が定義されている（BGMトラック・フェードアウト秒数・暗転秒数・noBlackout フラグ・typingRetry 演出・appIconFade・silenceDuration 等）。しかし UI設計書の「4-7. エンディング画面」では背景CSS1種・共通レイアウト・エンディング条件表のみを示しており、以下の差分が完全に未記述:
   - ズレエンド（zure）: `noBlackout: true`・BGMなし・フィード画面を明るいまま維持する演出
   - 既読エンド（kidoku）: フェードアウト最長12秒・`bgmDelay: 3000ms`・lastChoice の2択選択肢UI
   - 贖罪エンド（zange）: `typingText` の入力→消去→再入力アニメーション・karenTypingLoops 演出
   - 沈黙エンド（chinmoku）: `appIconFade`（アプリアイコンのフェードアウト）・`retryButtonDelay: 10000ms`・mysteryMessage（差出人不明メッセージ）の表示UI
   - 到達エンド（toutatu）: 「返信選択肢が出かけて消える」演出のUI定義
   これらは CSS だけでは表現しきれない JS 制御の演出であり、UI設計書に演出フローと対応するCSSクラス/アニメーション定義を追加する必要がある。

3. **キャラクター別スタイルに朔・誠司・花蓮のチャットスタイル差分が未定義**  
   「6. キャラクター別スタイル」では:
   - みどり: アバター背景・ボーダー・バブル背景・バブルボーダーが明記済み
   - 朔（6-2）: アバター背景が「共通（#2a4a2a, #1a3a1a）」とされているが、朔はパープル（#c084fc）がテーマカラーであり、共通の緑グラデーションアバターとの矛盾について説明がない
   - 誠司（6-3）・花蓮（6-4）: チャットスタイル（バブル背景・ボーダー色・アバター差分）の記述が完全に省略されている
   機能設計書では4キャラクターがテーマカラーを持つと定義されており、各キャラのチャット画面でテーマカラーをどう反映するかを明記する必要がある。

4. **mysteryClues 蓄積状態のUI反映方法が未記述**  
   機能設計書のセクション7「ミステリーシステム」では `GS.mysteryClues` が `karen`・`yukiwarisou` の2クルーを蓄積し、`mysteryClues.length >= 2` が花蓮ルートの謎収束選択肢出現条件であると定義している。UI設計書ではメモ帳ウィンドウ（`#memoAppWindow`）の記述はあるが、以下が未記述:
   - mysteryClues がメモ帳にどう表示されるか
   - クルー蓄積時のUI通知（トースト・アイコンバッジ等）
   - `mysteryClues.length >= 2` を満たした際に花蓮ルート中に選択肢が出現するトリガーのUI動作

## 改善推奨事項

5. **スコアバーの「スコア0〜100のどこで見た目が変わるか」が未定義**  
   機能設計書の「6. スコアシステム」では `buzz >= 65` や `self >= 60` などの閾値がエンディング分岐の境界値として定義されている。UI設計書では `.score-fill` の幅が0〜100%で変化することは自明だが、65%・60%・50% 等の閾値を超えたときに色や境界線が変化する仕様が存在するかどうか不明。閾値到達時の視覚フィードバック方針を明記すること。

6. **投稿ビルダーの「素材画像選択」UIが省略されている**  
   機能設計書セクション9で `GS.selectedImageSrc`（チャット中の画像も選択可能）が定義されているが、UI設計書「4-5. 投稿ビルダー画面」のレイアウト構造に画像選択UIが含まれていない。これが意図的な省略（画像選択は素材カード選択で代替）なのか、設計漏れなのかを明記する必要がある。

7. **フラッシュバック演出のUI詳細が不完全**  
   「5-7. フラッシュバック演出」では3段階の発火条件は記述されているが:
   - Phase 1（プロローグ）: 「CRT風5秒」の具体的なCSSクラス・演出定義がない（CRT フィルター等）
   - Phase 3（花蓮）: 「現在カードと過去記憶の2枚並列表示」の具体的なレイアウト実装方法が不明
   これらの実装仕様をUI設計書に追加する必要がある。

8. **seiji ルートで誠司の素材カードは3枚（他は4枚）だが言及なし**  
   機能設計書の素材リストを確認すると、みどり・朔は4枚、誠司は3枚、花蓮は3枚である。UI設計書「4-4. 素材確認画面」では `grid: repeat(2, 1fr)` のグリッドで説明しているが、カード枚数のルート差異についての説明がない（ロックカードで埋めるのか、グリッドが可変なのか）。

9. **エンディング画面のz-index 300 とチャトルウィンドウのz-index の整合性確認が必要**  
   UI設計書「3. 画面一覧」では `#screen-ending` の z-index を `300` と記述しているが、`#chatWindow` の z-index は `100〜` と曖昧な記述。既読エンド・贖罪エンドでは花蓮との「チャトル風」メッセージ演出があるため、エンディング画面上でチャトルUIを重ねて表示するのか否かを明確にする必要がある。

10. **キャラクター名の不一致がUI設計書内で未注記**  
    機能設計書セクション4の注記では「CHARACTERS オブジェクト（app-core.js）の名前（倉田 朔・松本 誠司）とマスター仕様書（水無月 朔・天羽 誠司）で名前が異なる」と明示されている。UI設計書セクション6では「水無月 朔」「天羽 誠司」を採用しているが、この選択が確定事項なのか、または未解決の課題なのかを追記すること。

## 軽微な指摘

11. **CSSとの軽微な齟齬: タイトル画面の `border-radius`**  
    UI設計書「2-3」のキャラクターカード（タイトル）の `border-radius` は `6px` と記述されているが、実際の CSS（`css/main.css` 内 `.ts-window`）では `border-radius: 8px` が設定されており1点の不一致がある。

12. **アニメーション定義表の `protagMoving` が実装と乖離**  
    UI設計書「2-4」の `protagMoving` アニメーションは `scale 1→1.08, rotate -4deg→4deg` と記述されているが、実際の CSS では `protagDragWobble`（`translateY 0→-4px, rotate -4deg→4deg`）が定義されており、`protagMoving` キーフレーム自体は CSS に存在しない可能性がある（検索でヒットしなかった）。アニメーション名と定義の照合が必要。

13. **「4-3. クイズ画面」の記述が実質的に空**  
    セクション「4-3. クイズ画面」は「専用の画面ではなくチャット画面の選択肢UIを活用」と説明するのみで、クイズ形式のUIコンポーネント詳細（正誤フィードバック・スコア加算の視覚化）が省略されている。機能設計書には `CHAT_QUIZ` データが存在するため、最低限のUI定義を追記すべき。

14. **セーブ/ロード画面のUI設計が欠落**  
    機能設計書「10. セーブ/ロード機能」では3スロットのセーブ・ロード・削除フローが定義されているが、UI設計書「3. 画面一覧」にはセーブ/ロード画面が含まれていない。タイトル画面内のドロワーとして実装されているとしても、スロット選択UI・削除確認ダイアログのコンポーネント設計を追加すること。

15. **`--win-header` 変数の「16161f → 202030」の役割記載を訂正**  
    UI設計書「2-1. カラーパレット」では `--win-header` の Hex 値が `#202030`・用途が「ウィンドウヘッダー背景」と記述されており、実際の CSS との一致は確認できた。しかし同表に `--surface` = `#16161f`（ウィンドウ表面）との関係性が読み取りにくく、ウィンドウの「ヘッダー（タイトルバー直下のサブヘッダー）」と「タイトルバー（`--win-titlebar: #141420`）」の使い分けについてコメントを追加すると可読性が向上する。

## 良かった点

1. **デザイントークンの網羅性が高く、CSSとの一致が精確**  
   `--bg`・`--surface`・`--accent` 系など主要CSS変数の Hex 値が CSS と完全一致しており、実装者がトークンを引用してそのまま利用できる状態になっている。

2. **アニメーション一覧（2-4）が実装リファレンスとして有用**  
   17種のキーアニメーションをパラメータ付きで一覧化しており、アニメーション管理の見通しが良い。特にトランジション基準値のコードブロック（5種）は実装ルールの統一に貢献している。

3. **チャットバブルの状態バリエーション（5-1）が丁寧に定義されている**  
   クライアント・プレイヤー・システムメッセージ・クリッパブル・クリップ済み・フラッシュバックの6状態がCSS付きで記述されており、チャットエンジンとの連携実装に有用。

4. **ウィンドウマネージャーのスナップ・最大化・リサイズの仕様が具体的**  
   3ゾーンスナップ・8方向リサイズハンドル・最小サイズ `280×200px` など、実装に必要な数値が揃っており、`window-manager.js` との対応が取りやすい。

5. **セクション「9. 今後のUI改善案」が機能設計書の課題と対応している**  
   機能設計書「11. 今後の実装課題」の未対応事項（アクセシビリティ・モバイル対応）を UI 設計書の改善案として引き継いでおり、両文書の継続的なメンテナンス連携が意識されている。
