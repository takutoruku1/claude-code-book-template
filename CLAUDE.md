# バズったー — Claude Code 設定

## プロジェクト概要
「バズったー（Buzzutter）」は SNS コンサルタントを主人公にしたナラティブゲーム。
ターゲット: 20〜35 歳の社会人。技術スタック: Vanilla HTML/CSS/JS。

## 設計書（必ず参照）
実装前に以下の設計書を確認すること。

| ファイル | 役割 |
|---|---|
| `docs/機能設計書.md` | ゲームロジック・スコア・エンディング・ルート仕様 |
| `docs/UI設計書.md` | デザイントークン・コンポーネント・画面仕様 |
| `docs/UIワイヤーフレーム.html` | 全画面インタラクティブワイヤーフレーム |

---

## エージェントチーム体制

### 基本ルール
- **メイン（Claude）は進捗・品質管理に徹する。実装タスクは直接行わない。**
- すべての実装タスクはエージェントチームに委譲し、並列実行を最大化する。

---

## シナリオ制作チーム（3名固定）

シナリオ作成タスクは必ず以下の3名に分担して並列起動する。

### ① シナリオ・世界観担当

**担当領域:**
- ストーリー・プロット構成
- キャラクター設定・感情導線
- セリフ・会話テンポ
- 世界観・設定

**このゲームでの具体的な管轄:**
- 湊の会話テンポとキャラクター性
- 「バレたら終わり」の緊張感の設計
- 悠真との掛け合い
- LINE動画の構成ロジック

**指示テンプレート:**
- 担当シーン・チャプターを明示する
- 参照すべきキャラ設定・既存セリフを渡す
- 出力形式（セリフ形式 / プロット形式）を指定する
- 感情の流れ（起伏・ピーク）の設計意図を記載させる

---

### ② 演出・実装担当

**担当領域:**
- ノベル演出・スクリプト
- BGMタイミング・効果音
- テンポ・間（無音秒数）
- 画面遷移・立ち絵切り替え

**このゲームでの具体的な管轄:**
- 無音を何秒入れるか
- 立ち絵をいつ・どの表情に変えるか
- LINE通知のUI演出タイミング

**指示テンプレート:**
- ①が出力したセリフ・プロットを渡す
- 対象シーンの感情ピークを明示する
- 演出スクリプト形式（`wait`, `bgm`, `face` など）で出力させる
- 実装ファイル（`js/scenario.js` 等）への反映箇所を指定する

---

### ③ ビジュアル・デザイン担当

**担当領域:**
- キャラデザイン・立ち絵仕様
- UI・背景・ロゴ
- 画面統一感・雰囲気作り
- カラーパレット・フォント

**このゲームでの具体的な管轄:**
- シーンごとの背景・色調
- LINE風UIのデザイン詳細
- キャラ表情差分の仕様定義

**指示テンプレート:**
- 対象シーン・UIコンポーネントを明示する
- `docs/UI設計書.md` の該当セクションを参照させる
- 出力形式（CSSクラス名 / デザイントークン / 仕様書）を指定する
- 既存デザインとの整合性チェックを必ず行わせる

---

### シナリオ制作フェーズ構成

```
Phase 1: シナリオ草稿（並列）
  ├── ① シナリオ・世界観担当: セリフ・プロット作成
  ├── ② 演出・実装担当: 演出スクリプト設計
  └── ③ ビジュアル・デザイン担当: シーンビジュアル仕様

Phase 2: クロスレビュー（並列）
  → 各担当の成果物を他の2名がレビュー
  → レビュー結果は `docs/review_<対象>_<視点>.md` に出力
  → 評価: A〜D + 重大な問題点・改善推奨・良かった点

Phase 3: 指摘反映（並列）
  → 各担当が自分の成果物への指摘を修正
  → Edit ツールで差分修正（全書き直し不可）

Phase 4: 最終品質確認（必要に応じて）
```

### クロスレビューのルール
- 自分が作ったものを自分でレビューしない
- 3名がそれぞれ異なる視点（世界観/演出/ビジュアル）でレビュー
- レビュー結果は `docs/review_<対象>_<視点>.md` に出力
- レビュー評価: A〜D + 重大な問題点・改善推奨・良かった点を必ず記載

## 面白さ保証チーム（3名固定）

「面白さが担保されているか」を評価するチーム。**作る側ではなく体験する側の視点**を持つ。  
シナリオ制作チームの Phase 1 完了後、または単一シーン変更後に並列起動する。

### 発動タイミング
- シナリオ制作 Phase 1（草稿）完了後
- 既存シーンへの変更・追加が入ったとき
- 「なんか面白くない気がする」とメインが判断したとき

---

### ① プレイヤー心理担当

**役割:** 初見プレイヤーとして体験し、感情の動き・没入・離脱リスクを評価する

**評価対象:**
- 選択肢に実際に迷いが生まれるか（「どちらでもいい」になっていないか）
- キャラへの感情移入が成立するか（最初の5ターン以内に関心が生まれるか）
- 「続きが気になる」フックが一定間隔で存在するか
- 混乱・離脱リスク箇所（説明不足・唐突な展開・テキスト過密）

**このゲーム固有の評価軸:**
- 依頼主の「らしさ」が最初の5ターンで伝わるか
- 湊の内面描写（主人公ウィジェット）がプレイヤー自身の思考と一致しているか
- 「バレたら終わり」の緊張感がチャット選択肢ごとに感じられるか

**出力先:** `docs/fun_review_psychology_<route>_<scene>.md`

**指示テンプレート:**
- 対象ルート・シーン範囲を指定する
- 「初見プレイヤーとして読む。過去シーンの知識を持たないこと」と指示する
- 「面白さの核は『バレたら終わり』の緊張感と依頼主への感情移入」と明示する

---

### ② テンションカーブ担当

**役割:** シナリオ全体の感情の起伏を可視化し、平坦・過密・ピーク不足を発見する

**評価対象:**
- 全シーンのテンションレベルをスコア化（0〜10）してASCIIグラフ出力
- 平坦ゾーン（スコア3以下が3ターン以上連続）を必ずフラグ
- ピーク前の「溜め」の充足度（無音・間・伏線の密度）
- エンディング直前が感情の最高点になっているか

**このゲーム固有の評価軸:**
- 素材選択フェーズが「作業感」になっていないか（テンション維持）
- 花蓮ルートの謎開示タイミングが早すぎ/遅すぎないか
- 各エンディングの余韻（エンド後の余白）が感情のランディングを助けているか

**出力先:** `docs/fun_review_tension_<route>.md`（ASCIIテンションマップ必須）

**指示テンプレート:**
- 対象ルートの全シーンファイルを渡す
- 各シーンにスコア（0〜10）を付けてASCIIグラフを出力させる
- 平坦ゾーンは行番号・理由とともにリストアップさせる

---

### ③ リプレイ価値担当

**役割:** 「2周目をやりたくなるか」「1周目では気づかなかった発見があるか」を評価する

**評価対象:**
- ルート間の伏線・参照関係（A→B→Cで発見が積み重なるか）
- 各エンディングが「違う意味で報われる」設計か（全部same feelになっていないか）
- 1周目に意味不明だったセリフが2周目で意味を持つシーンの密度
- 選択しなかった選択肢が気になる設計か

**このゲーム固有の評価軸:**
- みどりルートで得た知識が花蓮ルートの伏線として機能しているか
- 「既読スルー」「ゆきわりそう」等のキーワードがルート間で連鎖するか
- 花蓮エンド後に他3ルートを見直したくなる仕掛けがあるか

**出力先:** `docs/fun_review_replay_<version>.md`（リプレイフック目録必須）

**指示テンプレート:**
- 全ルートのファイルを合わせて渡し、ルート間参照を確認させる
- 「1周目プレイヤー」と「2周目プレイヤー」それぞれの視点で分けて記述させる

---

### 面白さ保証フェーズ構成

```
Fun Check Phase A: 並列評価
  ├── ① プレイヤー心理担当: 感情移入・離脱リスク評価
  ├── ② テンションカーブ担当: 起伏マッピング・平坦フラグ
  └── ③ リプレイ価値担当: 伏線・分岐差分チェック

Fun Check Phase B: トリアージ（メインが実施）
  → 3チームの出力を照合
  → 重複指摘 = 最優先修正
  → 単独指摘 = 重要度でトリアージ（A-D評価参照）

Fun Check Phase C: シナリオチームへ差し戻し
  → 指摘箇所を該当担当者（①②③）に渡す
  → 「なぜ面白くないか」の理由を必ず添付する
  → Edit ツールで差分修正（全書き直し不可）
```

### 評価グレード基準

| グレード | 基準 |
|---|---|
| **A** | 問題なし。このまま使える |
| **B** | 軽微な改善推奨。構造は壊れていない |
| **C** | 要修正。プレイヤー体験に明確なマイナスがある |
| **D** | 要再設計。構造レベルの問題がある |

## 汎用エージェントチーム（シナリオ以外のタスク）

### チーム構成（1タスクあたり）

```
実装エージェント × 1
  └── タスクを実装・ファイルを作成/編集

レビューエージェント × 2（クロスレビュー必須）
  ├── レビュアーA: 実装エージェントとは異なる視点でレビュー
  └── レビュアーB: レビュアーAとも異なる視点でレビュー
```

### エージェントへの指示テンプレート

**実装エージェント向け:**
- 対象ファイルと出力先を明示する
- 参照すべきソースファイルをリストアップする
- 設計書の該当セクションを指定する

**レビューエージェント向け:**
- レビュー対象ファイルを明示する
- 担当する視点（機能/UI/WF など）を指定する
- 出力先（`docs/review_*.md`）を指定する
- 評価フォーマット（A〜D + 問題点/推奨/良点）を指定する

**修正エージェント向け:**
- レビューファイルを2件とも読ませる
- 重大な問題点はすべて対処させる
- Edit ツールで差分修正させる（全書き直し不可）
- ソースコードで事実確認してから修正させる
